## SECTION 1 — THE UNIVERSAL SOP TEMPLATE

**Header block:**
"SOP Title: [Process name] | Owner: [Name/role] | Last updated: [date] | Applies to: [team/role/situation]"

**Body — 5 sections every SOP needs:**
1. **Purpose** — Why this process exists (1-2 lines — the "why," not just the "what")
2. **When to use this** — Trigger/scenario that kicks off the process
3. **Step-by-step instructions** — Numbered, specific, written for someone doing it for the first time
4. **Common issues + fixes** — What usually goes wrong, and how to handle it
5. **Who to contact if stuck** — Clear escalation point

**Step-writing rule:** Each step = one action + the expected result ("Click [X] → you should see [Y]"). If a step requires a decision, spell out the decision criteria clearly.

## SECTION 5 — DOCUMENTATION MAINTENANCE SYSTEM

**The "update trigger" rule:** Every time someone asks a question that SHOULD be documented, that's the trigger to write/update the doc — not "when I have time" (which never comes).

**Quarterly review checklist:**
- Which docs haven't been opened/used in 90 days? (candidates for archiving or rewriting)
- Which questions came up repeatedly that aren't documented yet?
- Which SOPs changed in practice but not on paper? (biggest source of confusion)

## 6 RULES FOR DOCUMENTATION THAT PEOPLE ACTUALLY USE
1. Write for someone doing this for the FIRST time — not for someone who already knows it (that's who needs it most)
2. Organize by "what someone is trying to do," not by department structure — people search by task, not org chart
3. Keep steps specific and sequential — vague instructions create more questions than they answer
4. Update docs the moment a process changes — outdated documentation is worse than no documentation (it actively misleads)
5. Make docs easy to find — a perfect SOP that's buried three folders deep helps no one
6. Treat "I keep explaining this" as a signal — that's exactly what should be documented next
